# 🛒 POS — ระบบขายหน้าร้าน

ระบบ Point of Sale ครบวงจร ใช้งานง่าย รองรับทั้ง **มือถือ / iPad / PC** (responsive)
สร้างด้วย **Next.js + Tailwind CSS + Supabase** พร้อม deploy บน **Vercel** ได้ทันที

## ✨ ฟีเจอร์

- **ขายหน้าร้าน (POS)** — 2 โหมดแยกกันชัดเจน: **ขายด่วน** (เลือกสินค้า → คิดเงินจบทันที เหมาะกับซื้อกลับบ้าน) และ **เปิดโต๊ะ** (กินก่อนจ่ายทีหลัง — เลือกโต๊ะ สั่งอาหารได้หลายรอบ ระบบรวมเป็นบิลเดียวของโต๊ะอัตโนมัติ **แก้ไข/ลบรายการที่สั่งไปแล้วได้ถ้ากดผิด** ค่อยเก็บเงิน/ปิดโต๊ะตอนจบมื้อ) ทั้งสองโหมดค้นหา/สแกนบาร์โค้ด, กรองตามหมวดหมู่, ส่วนลด, ชำระด้วยเงินสด/โอน/บัตร, คำนวณเงินทอนอัตโนมัติ
- **บันทึกยอดขาย** — ทุกบิลถูกบันทึกพร้อมรายการสินค้า ราคา และต้นทุน ณ เวลาขาย, ดูประวัติย้อนหลังตามช่วงวันที่, ยกเลิกบิล (คืนสต๊อกอัตโนมัติ)
- **บันทึกต้นทุน & กำไร** — กำหนดต้นทุนต่อชิ้นของสินค้า ระบบคำนวณกำไรต่อชิ้น กำไรต่อบิล และกำไรรวมให้อัตโนมัติ
- **บันทึกรายจ่าย/ต้นทุนกิจการ** — บันทึกต้นทุนเริ่มต้น/เงินลงทุน, วัตถุดิบ, ค่าเช่า, ค่าน้ำ-ค่าไฟ, เงินเดือน, ค่าขนส่ง, การตลาด ฯลฯ แยกตามหมวดหมู่ที่ปรับแต่งเองได้ ค้นหา/กรองตามหมวดหมู่/ช่วงวันที่
- **บันทึกรายได้อื่นๆ** — รายได้นอกเหนือจากยอดขาย POS เช่น เงินลงทุนเพิ่ม, รายได้ค่าบริการ, เงินคืนจากซัพพลายเออร์ — ค้นหา/กรอง/สรุปตามหมวดหมู่เหมือนหน้ารายจ่าย พร้อม **บันทึกยอดขายเองได้** สำหรับยอดขายนอกระบบ POS (เช่น ขายผ่านโทรศัพท์/แอปส่งของ) ซึ่งจะถูกนับรวมเป็นยอดขายจริงในหน้าภาพรวมและประวัติการขายทันที
- **จัดการสินค้า & สต๊อก** — เพิ่ม/แก้ไข/ลบสินค้า, หมวดหมู่, บาร์โค้ด, ตัดสต๊อกอัตโนมัติเมื่อขาย, แจ้งเตือนสินค้าใกล้หมด
- **Dashboard ภาพรวมร้าน** — ยอดขาย ต้นทุนสินค้า กำไรขั้นต้น รายได้อื่นๆ รายจ่ายอื่นๆ **กำไรสุทธิ** จำนวนบิล, กราฟยอดขายรายวัน, สินค้าขายดี Top 5, แยกยอดตามวิธีชำระเงิน, แยกรายจ่ายตามหมวดหมู่ (วันนี้ / 7 วัน / 30 วัน / เดือนนี้)
- **ระบบ role พนักงาน** — พนักงานขายและผู้จัดการเข้าถึงเท่ากัน (ขายหน้าร้าน/ประวัติขาย/จัดการสินค้า) ส่วนภาพรวมร้าน/รายได้/รายจ่าย/จัดการพนักงาน สงวนไว้ให้ **เจ้าของร้าน** เท่านั้น — บังคับสิทธิ์จริงที่ระดับฐานข้อมูล (RLS) ไม่ใช่แค่ซ่อนเมนู
- **สร้างพนักงานในแอปได้เลย + ล็อกอินด้วยเบอร์โทร** — เจ้าของร้านกด "เพิ่มพนักงาน" ตั้งเบอร์โทร+รหัสผ่านให้เอง (ไม่ต้องมีอีเมลก็สร้างได้) พนักงานใช้ **เบอร์โทรหรืออีเมล + รหัสผ่าน** ล็อกอินได้ทันที และเจ้าของร้านรีเซ็ตรหัสผ่านให้พนักงานได้ตลอดเวลา
- **Log กิจกรรม** — บันทึกทุกการเพิ่ม/แก้ไข/ลบในระบบอัตโนมัติเป็นข้อความอ่านง่าย (สินค้า, หมวดหมู่, รายรับ-รายจ่าย, โต๊ะ, พนักงาน, การขาย/ยกเลิกบิล/แก้ไขออเดอร์) พร้อมชื่อผู้ทำรายการและเวลา — หน้า "Log" เห็นได้เฉพาะเจ้าของร้าน ค้นหา/กรองตามช่วงวันที่ได้
- **ลูกค้าสแกน QR สั่งอาหารเอง** — แต่ละโต๊ะมี QR โค้ดของตัวเอง (สร้าง/ดาวน์โหลดได้จากหน้า "จัดการโต๊ะ") ลูกค้าสแกนแล้วสั่งอาหารได้เลย **ไม่ต้องล็อกอิน** ดูออเดอร์/ยอดรวมของตัวเองได้แบบเรียลไทม์ กด **"เรียกเก็บเงิน"** เพื่อแจ้งพนักงานได้ (ขึ้นแจ้งเตือนในหน้าเปิดโต๊ะฝั่งพนักงานทันที) พนักงานปิดบิลเสร็จโต๊ะนั้นจะว่างพร้อมรับลูกค้าใหม่โดยอัตโนมัติ — ต้นทุนสินค้าถูกซ่อนจากลูกค้าเสมอ
- **ระบบล็อกอิน** — ผ่าน Supabase Auth ปลอดภัยด้วย Row Level Security

## 🧰 Tech Stack

| ส่วน | เทคโนโลยี |

| Frontend | Next.js 14 (App Router) + TypeScript |
| UI | Tailwind CSS (responsive: mobile / tablet / desktop) |

## 🚀 วิธีติดตั้ง

### 1. สร้างโปรเจกต์ Supabase

1. ไปที่ [supabase.com](https://supabase.com) → **New Project**
2. เมื่อสร้างเสร็จ ไปที่ **SQL Editor** → วางเนื้อหาทั้งหมดจากไฟล์ [`supabase/schema.sql`](supabase/schema.sql) → กด **Run**
3. ไปที่ **Authentication → Users → Add user** → สร้างบัญชีเจ้าของร้าน (อีเมล + รหัสผ่าน) — แนะนำติ๊ก *Auto Confirm User*
   - **บัญชีแรกที่สมัครจะได้สิทธิ์ "เจ้าของร้าน" อัตโนมัติ** บัญชีถัดไปจะเป็น "พนักงานขาย" โดยปริยาย — ไปปรับ role ได้ที่เมนู "พนักงาน" ในแอป (เจ้าของร้านเท่านั้นที่เห็นเมนูนี้)
4. ไปที่ **Settings → API** จด 3 ค่านี้ไว้:
   - `Project URL`
   - `anon public` key
   - `service_role` key (กด "Reveal" เพื่อดู) — **เก็บเป็นความลับ ห้ามใส่ในโค้ดฝั่งเบราว์เซอร์หรือ commit ขึ้น git เด็ดขาด** ใช้สำหรับให้เจ้าของร้านสร้างบัญชีพนักงาน/รีเซ็ตรหัสผ่านจากในแอปเท่านั้น

### 2. รันบนเครื่อง (local)

```bash
git clone https://github.com/NTPxz/POS.git
cd POS
npm install
cp .env.example .env.local   # แล้วแก้ไขใส่ค่าจาก Supabase (รวม service_role key)
npm run dev
```

เปิด http://localhost:3000 แล้วล็อกอินด้วยบัญชีที่สร้างไว้

### 3. Deploy ขึ้น Vercel

1. ไปที่ [vercel.com](https://vercel.com) → **Add New Project** → เลือก repo `NTPxz/POS`
2. ที่หน้า config ใส่ **Environment Variables**:
   - `NEXT_PUBLIC_SUPABASE_URL` = Project URL ของ Supabase
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY` = anon key ของ Supabase
   - `SUPABASE_SERVICE_ROLE_KEY` = service_role key ของ Supabase (**ห้ามติ๊ก "Expose to browser"** — ค่านี้ต้องอยู่ฝั่งเซิร์ฟเวอร์เท่านั้น ปกติ Vercel จะไม่ expose ให้อยู่แล้วตราบใดที่ชื่อตัวแปรไม่ขึ้นต้นด้วย `NEXT_PUBLIC_`)
3. กด **Deploy** — เสร็จแล้ว! เปิดใช้ได้จากมือถือ / iPad / PC ได้เลย

> 💡 เปิดจากมือถือแล้วกด "Add to Home Screen" จะใช้งานได้เหมือนแอปเลย

### 4. ตั้งค่าแจ้งเตือนแม้ปิดหน้าจอ (Web Push) — ไม่บังคับ

ฟีเจอร์แจ้งเตือนเสียง + toast ในเว็บ (ดูหัวข้อ "แจ้งเตือนพนักงานแบบ real time" ด้านล่าง) ใช้ได้ทันทีไม่ต้องตั้งค่าอะไรเพิ่ม แต่ถ้าอยากให้แจ้งเตือนได้แม้ **ล็อกจอ/สลับแอป/ปิดแท็บ** ต้องตั้งค่า Web Push เพิ่ม:

1. สร้างคีย์: `npx web-push generate-vapid-keys` แล้วเติมลง Environment Variables ทั้งใน `.env.local` (local) และ Vercel (production):
   - `NEXT_PUBLIC_VAPID_PUBLIC_KEY`
   - `VAPID_PRIVATE_KEY` (**ห้าม** ติ๊ก expose to browser)
   - `VAPID_SUBJECT` = `mailto:อีเมลของคุณ`
   - `PUSH_TRIGGER_SECRET` = สุ่มค่าเอง เช่น `openssl rand -hex 24` (ห้าม expose to browser)
2. เปิด Supabase Dashboard → SQL Editor → รันไฟล์ `supabase/migrations_manual/add_push_notifications.sql` — แก้บรรทัด `push_notify_url` ให้เป็นโดเมนจริงของคุณ และ `push_trigger_secret` ให้ตรงกับค่า `PUSH_TRIGGER_SECRET` ที่ตั้งไว้ในข้อ 1 (ไฟล์นี้เปิด extension `pg_net` เพื่อให้ฐานข้อมูลยิง HTTP กลับมาแจ้งเว็บได้ จึงต้องรันเองผ่าน Dashboard)
3. Deploy ใหม่อีกครั้งให้ env vars มีผล — จากนั้นพนักงานที่ล็อกอินจะเห็นแบนเนอร์ "เปิดแจ้งเตือนแม้ปิดหน้าจอ" ให้กดอนุญาตได้เอง (ต้องกดยินยอมทีละเครื่อง/เบราว์เซอร์)

    types.ts               # Type ของข้อมูลทั้งหมด
    format.ts              # ฟังก์ชันจัดรูปแบบเงิน/วันที่
  components/

    ProfileProvider.tsx    # โหลด role ของผู้ใช้ปัจจุบัน แชร์ผ่าน context
    RequireRole.tsx        # กั้นการเข้าถึงหน้าตาม role ขั้นต่ำ
  app/
    login/                 # หน้าล็อกอิน (อีเมลหรือเบอร์โทร)
    order/[tableId]/       # หน้าลูกค้าสั่งอาหาร (public, ไม่ต้องล็อกอิน) — สแกน QR เข้ามาถึง
    api/staff/
      create/route.ts       # owner สร้างบัญชีพนักงาน (ใช้ service_role)

    (app)/
      page.tsx             # หน้าขายสินค้า (POS) — สลับโหมด "ขายด่วน" / "เปิดโต๊ะ" — ทุก role เข้าได้

      sales/               # ประวัติการขาย + ยกเลิกบิล (ทุก role)

      expenses/            # รายจ่าย/ต้นทุนกิจการ + หมวดหมู่รายจ่าย (เจ้าของร้านเท่านั้น)
      products/            # จัดการสินค้า หมวดหมู่ สต๊อก (ทุก role)
      staff/                # เพิ่มพนักงาน, จัดการ role, รีเซ็ตรหัสผ่าน (เจ้าของร้านเท่านั้น)
      log/                  # Log กิจกรรมทั้งระบบ (เจ้าของร้านเท่านั้น)
  components/pos/
    ProductPicker.tsx      # ค้นหา/กรองหมวดหมู่/กริดสินค้า (ใช้ร่วมทั้ง 2 โหมด)
    PaymentFields.tsx      # เลือกวิธีชำระ + คำนวณเงินทอน (ใช้ร่วมทั้ง 2 โหมด)
    QuickSaleView.tsx      # โหมดขายด่วน — ตะกร้า + คิดเงินจบทันที
    TablesView.tsx         # โหมดเปิดโต๊ะ — กริดโต๊ะ, สั่งหลายรอบรวมบิลเดียว, เก็บเงิน/ปิดโต๊ะ, จัดการโต๊ะ

## 🗄️ โครงสร้างฐานข้อมูล

- `categories` — หมวดหมู่สินค้า
- `products` — สินค้า (ราคาขาย, ต้นทุน, สต๊อก, บาร์โค้ด)
- `dining_tables` — โต๊ะในร้าน (เจ้าของร้านตั้งค่าไว้ล่วงหน้า)
- `sales` — บิลขาย (ยอดรวม, ส่วนลด, ต้นทุนรวม, วิธีชำระ, เงินรับ/ทอน, สถานะ `open`/`completed`/`voided`, `table_id` ถ้ามาจากการเปิดโต๊ะ, `customer_name` ชื่อลูกค้าใส่ได้ทั้งหน้าโต๊ะและหน้าประวัติขาย ไม่บังคับ)
- `sale_items` — รายการสินค้าในบิล (บันทึกราคาและต้นทุน ณ เวลาขาย, `ordered_by` staff/customer, `status` pending/accepted/served สำหรับติดตามสถานะครัว)
- `expense_categories` / `expenses` — หมวดหมู่และรายการรายจ่าย/ต้นทุนกิจการ (มี `payment_method` cash/transfer/card แยกวิธีจ่ายด้วย)
- `income_categories` / `income` — หมวดหมู่และรายการรายได้อื่นนอกเหนือจากยอดขาย POS
- `profiles` — role + เบอร์โทรของผู้ใช้แต่ละคน (`owner` / `manager` / `staff`) สร้างอัตโนมัติเมื่อสมัครผ่าน trigger, เบอร์โทร unique
- `activity_log` — log กิจกรรมทุกการเพิ่ม/แก้ไข/ลบในระบบ (เห็นได้เฉพาะ owner)
- `public_menu` — view สำหรับลูกค้า (ซ่อนคอลัมน์ `cost` แทนด้วย 0) ให้ `anon` อ่านได้โดยไม่ต้องล็อกอิน
- `promotions` — โปรโมชั่นที่เจ้าของร้านตั้งไว้ (`name`, `type`, `threshold_qty`, `discount_amount`, `is_active`) — owner จัดการเท่านั้น, `anon`/`authenticated` อ่านได้เฉพาะที่ `is_active = true`
- `promotion_products` — เมนูที่เข้าร่วมแต่ละโปรโมชั่น (เจ้าของร้านเลือกเอง) — สิทธิ์เข้าถึงตามโปรโมชั่นแม่เหมือนกัน
- `cash_shifts` — กะเงินสดหน้าร้าน (ตั้งต้น, ปิดกะ/นับเงินจริง, ยอดคาดการณ์, ส่วนต่าง) พนักงานทุกคนเข้าถึงได้เหมือน `sales`
- `account_adjustments` — รายการปรับยอดด้วยมือ (`account` cash/transfer, จำนวนเงิน +/-, เหตุผล) ผูกกับกะได้ถ้าเป็นการปรับเงินสดระหว่างกะ
- `announcements` — ประกาศจากเจ้าของร้านถึงพนักงาน (ข้อความ ≤ 500 ตัวอักษร) — owner ส่ง/ลบได้เท่านั้น, `authenticated` ทุกคนอ่านได้ (ใช้ Realtime แจ้งแบบสด)
- ฟังก์ชัน `create_sale()` — บันทึกบิล "ขายด่วน" + ตัดสต๊อกใน transaction เดียว (กันข้อมูลเพี้ยน)
- ฟังก์ชัน `add_order_to_table()` — เปิดบิล "open" ใหม่ให้โต๊ะ หรือรวมรายการเข้าบิล "open" เดิมถ้ามีอยู่แล้ว (กินหลายรอบ = บิลเดียว) — พนักงานเรียกใช้
- ฟังก์ชัน `customer_add_order()` — เหมือน `add_order_to_table()` แต่เปิดให้ `anon` เรียกได้ (ลูกค้าสั่งเอง) ตรวจสอบโต๊ะมีจริง/เปิดใช้งานก่อนเสมอ จำกัดจำนวนต่อรายการไม่เกิน 99
- ฟังก์ชัน `get_table_order()` — คืนออเดอร์ปัจจุบันของโต๊ะแบบไม่โชว์ต้นทุน (ลูกค้า/พนักงานเรียกดูได้)
- ฟังก์ชัน `request_checkout()` — ลูกค้ากด "เรียกเก็บเงิน" ปักธง `bill_requested_at` ให้พนักงานเห็น
- ฟังก์ชัน `cancel_bill_request()` — พนักงานกดยกเลิกการเรียกเก็บเงิน (เผื่อลูกค้ากดผิด) เคลียร์ `bill_requested_at` กลับเป็น null
- ฟังก์ชัน `update_table_order_item()` — แก้ไขจำนวน/ลบรายการที่สั่งไปแล้วในบิลที่ยังเปิดอยู่ (กันกดผิด) ปรับสต๊อก/ยอดบิลให้อัตโนมัติ
- ฟังก์ชัน `update_sale_item_status()` — เปลี่ยนสถานะรายการ (`pending` → `accepted` → `served`) สำหรับติดตามความคืบหน้าครัว พนักงานเท่านั้นที่เรียกได้ (ลูกค้าปิดสถานะออเดอร์ตัวเองไม่ได้)
- ฟังก์ชัน `calculate_promo_discount()` — คำนวณส่วนลดโปรโมชั่นทั้งหมดที่ใช้ได้กับบิล เรียกอัตโนมัติจาก `add_order_to_table()`/`customer_add_order()`/`update_table_order_item()` ทุกครั้งที่รายการเปลี่ยน
- ฟังก์ชัน `open_cash_shift()` / `close_cash_shift()` — เปิด/ปิดกะเงินสด ปิดกะจะคำนวณยอดคาดการณ์ (ตั้งต้น + ยอดขายเงินสด − รายจ่ายเงินสด + ยอดปรับระหว่างกะ นับตั้งแต่เปิดกะ) เทียบกับยอดนับจริงให้อัตโนมัติ
- ฟังก์ชัน `add_cash_adjustment()` / `add_transfer_adjustment()` — ปรับยอดเงินสด/เงินโอนด้วยมือ (บวกหรือลบก็ได้) พร้อมบังคับใส่เหตุผลเสมอ ปรับเงินสดได้เฉพาะตอนกะยังเปิดอยู่
- trigger `notify_push_on_announcement()` — ยิง Web Push ให้พนักงานทุกครั้งที่มีประกาศใหม่จากเจ้าของร้าน ผ่านกลไก `trigger_push_notify()` เดียวกับออเดอร์/เรียกเก็บเงิน
- ฟังก์ชัน `checkout_table()` — ปิดบิล "open" เป็น "completed" พร้อมส่วนลด/วิธีชำระ/เงินรับ-ทอน
- ฟังก์ชัน `void_sale()` — ยกเลิกบิล (รองรับทั้งบิล `open` และ `completed`) + คืนสต๊อก
- ฟังก์ชัน `my_role()` / `is_manager_or_above()` / `is_owner()` — ใช้ตรวจสิทธิ์ใน RLS policy ของทุกตาราง
- ฟังก์ชัน `get_email_by_phone()` — ค้นหาอีเมลจากเบอร์โทรตอนล็อกอิน (normalize ตัวเลขทั้งสองฝั่งให้ทนเว้นวรรค/ขีด)
- ฟังก์ชัน `log_action()` — เขียน log แบบข้อความอ่านง่ายจาก RPC ต่างๆ (ขาย/ยกเลิก/สั่งอาหาร/แก้ไขออเดอร์)
- trigger `log_activity()` — log อัตโนมัติสำหรับตารางที่แก้ไขตรงจากหน้าเว็บ (สินค้า, หมวดหมู่, รายรับ-รายจ่าย, โต๊ะ, พนักงาน)

### โหมด "เปิดโต๊ะ" (กินก่อนจ่ายทีหลัง) ทำงานอย่างไร

1. เจ้าของร้านตั้งจำนวน/ชื่อโต๊ะไว้ล่วงหน้าที่ปุ่ม "จัดการโต๊ะ" ในโหมดเปิดโต๊ะ
2. พนักงานแตะเลือกโต๊ะที่ลูกค้านั่ง → สั่งอาหาร → กด "ส่งออเดอร์" (เรียก `add_order_to_table`) — ทำซ้ำได้เรื่อยๆ ตลอดมื้อ ทุกรอบจะถูกรวมเป็นบิล `open` เดียวของโต๊ะนั้นอัตโนมัติ
3. กดผิด/สั่งผิดได้ — รายการ "สั่งไปแล้ว" แก้จำนวนหรือลบออกได้ทันทีถ้าบิลยังเปิดอยู่ (สต๊อก/ยอดบิลปรับให้อัตโนมัติ)
4. เมื่อลูกค้าพร้อมจ่าย กด "เก็บเงิน / ปิดโต๊ะ" → ใส่ส่วนลด/วิธีชำระ/เงินรับ → บิลเปลี่ยนเป็น `completed` โต๊ะกลับมาว่างพร้อมรับลูกค้าใหม่
5. บิล `open` (โต๊ะที่ยังไม่ปิด) จะไม่แสดงในหน้า "ประวัติการขาย" หรือนับรวมในยอดขาย/กำไรจนกว่าจะปิดโต๊ะจริง
6. ใส่/แก้ชื่อลูกค้าประจำบิลได้ทั้งจากหน้าโต๊ะ (ระหว่างเปิดบิลอยู่) และหน้า "ประวัติการขาย" (แก้ย้อนหลังได้แม้บิลปิดไปแล้ว) — ช่วยให้จำบิลง่ายขึ้นตอนมีลูกค้าหลายโต๊ะพร้อมกัน

### ลูกค้าสแกน QR สั่งอาหารเองทำงานอย่างไร

1. ในหน้า "จัดการโต๊ะ" (owner) กดไอคอน QR ข้างชื่อโต๊ะ → ได้ QR โค้ดที่ชี้ไปที่ `https://<โดเมนคุณ>/order/<table-id>` → ดาวน์โหลดไปพิมพ์ติดโต๊ะ
2. ลูกค้าสแกน → เจอเมนู (จากตาราง/view `public_menu` ที่ไม่มี `cost`) → เลือกสินค้า → กด "สั่งอาหาร" (เรียก `customer_add_order`, **ไม่ต้องล็อกอิน**) — ใช้ RPC เดียวกันกับที่พนักงานใช้เกือบทั้งหมด รวมเข้าบิล `open` เดียวกันกับที่พนักงานอาจเปิดไว้แล้ว
3. ลูกค้าเห็นออเดอร์/ยอดรวมของตัวเอง รีเฟรชอัตโนมัติทุก ~12 วินาที กด "เรียกเก็บเงิน" ได้เมื่อพร้อมจ่าย (เรียก `request_checkout`)
4. ฝั่งพนักงานในหน้าเปิดโต๊ะจะเห็น badge "เรียกเก็บเงิน" สีเหลืองบนโต๊ะนั้นทันที ไปกดปิดโต๊ะ/เก็บเงินได้ตามปกติ — ถ้าเข้าไปดูรายละเอียดโต๊ะจะเห็นแถบเหลือง "ลูกค้ากดเรียกเก็บเงินแล้ว" พร้อมปุ่ม **"ยกเลิก (ลูกค้ากดผิด)"** ไว้กดยกเลิกได้ทันทีถ้าลูกค้ากดผิด (เรียก `cancel_bill_request`) โดยไม่ต้องปิดบิลจริง
5. **ความปลอดภัย**: ลูกค้า (`anon`) เข้าถึงได้แค่ 3 ฟังก์ชันที่ควบคุมไว้ (`customer_add_order`, `get_table_order`, `request_checkout`) และอ่านได้แค่ `public_menu`/`categories`/`dining_tables` เท่านั้น — เข้าตาราง `sales`/`sale_items`/`products` ตรงๆ ไม่ได้เลย ทดสอบยืนยันด้วย SQL แล้วว่า anon อ่าน `sales` ตรงไม่ได้จริง
6. **แจ้งเตือนพนักงานแบบ real time**: ทุกครั้งที่ลูกค้าสั่งอาหารหรือกดเรียกเก็บเงิน หน้าเว็บของพนักงาน/ผู้จัดการ/เจ้าของร้าน (ทุกหน้า ไม่ใช่แค่หน้าโต๊ะ) จะเด้งแจ้งเตือนพร้อมเสียงทันทีผ่าน Supabase Realtime — `sale_items.ordered_by` แยกว่ารายการนั้นลูกค้าสั่งเอง (`'customer'`) หรือพนักงานคีย์ให้ (`'staff'`) เพื่อไม่ให้แจ้งเตือนซ้ำเวลาพนักงานคีย์ออเดอร์เอง
   - เสียงเป็นทำนองสังเคราะห์เอง (Web Audio API) ยาวรวม ~10 วิ ดังชัดเจน ไม่ต้องใช้ไฟล์เสียง
   - ระหว่างล็อกอินอยู่ หน้าเว็บจะกันจอดับอัตโนมัติด้วย Screen Wake Lock API จะได้ไม่พลาดเสียงแจ้งเตือนเพราะจอปิดไปเอง (จะปลดล็อกก็ต่อเมื่อพนักงานล็อกจอเองหรือสลับแอป)
   - ถ้าตั้งค่า Web Push ไว้ (ดูขั้นตอนที่ 4 ด้านบน) จะได้รับแจ้งเตือนแบบ OS-level แม้ปิดแท็บ/ล็อกจอ/สลับแอปไปแล้วด้วย — DB trigger บน `sale_items`/`sales` ยิง webhook ไปที่ `/api/push/notify` (ยืนยันด้วย secret header) แล้ว route นั้นส่ง push ผ่าน `web-push`/VAPID ไปยังทุกอุปกรณ์ที่เคยกด "เปิดแจ้งเตือน" ไว้ (เก็บ endpoint ไว้ในตาราง `push_subscriptions` ผูกกับ user คนนั้น ลบอัตโนมัติถ้า endpoint หมดอายุ)
   - นอกจาก toast/เสียงชั่วคราวแล้ว ยังมีจุดแดงค้างอยู่ที่ไอคอน "ขายสินค้า" บนเมนู (ทุกหน้า) จนกว่าจะกดเข้าไปดู — กดแล้วพาไปโหมด "เปิดโต๊ะ" ให้อัตโนมัติ
   - toast ของออเดอร์ใหม่/เรียกเก็บเงินกดได้เจาะจงเลยว่าโต๊ะไหน — กดแล้วพาเข้าไปหน้าโต๊ะนั้นตรงๆ ทันที ไม่ต้องไปหาเองในหน้ารายการโต๊ะ
   - ถ้ารายการไหนยังสถานะ `pending` ("รอรับ") ค้างเกิน 1 นาที ระบบจะเตือนซ้ำเป็นระยะ (toast สีแดง + เสียงเตือนคนละแบบกับออเดอร์ใหม่) จนกว่าจะกด "รับออเดอร์" จริงถึงจะหยุด — ในหน้าเปิดโต๊ะแต่ละรายการที่ยัง pending จะมีตัวนับถอยหลังให้เห็นด้วยว่าใกล้โดนเตือนซ้ำแค่ไหน

### ระบบติดตามสถานะครัว (รับออเดอร์ → เสิร์ฟแล้ว) ทำงานอย่างไร

1. ทุกรายการที่สั่งเข้าโต๊ะ (ไม่ว่าพนักงานคีย์เองหรือลูกค้าสั่งผ่าน QR) เริ่มต้นที่สถานะ `pending` ("รอรับออเดอร์")
2. ในหน้าเปิดโต๊ะ พนักงาน/ครัวกด **"รับออเดอร์"** ที่รายการนั้น → เปลี่ยนเป็น `accepted` ("กำลังเตรียม") เรียก `update_sale_item_status`
3. พอทำเสร็จเสิร์ฟให้ลูกค้าแล้ว กด **"เสิร์ฟแล้ว"** → เปลี่ยนเป็น `served` ("เสิร์ฟแล้ว") ถือว่ารายการนั้นจบงาน
4. การ์ดโต๊ะในหน้าเลือกโต๊ะจะมีตัวเลขสีแดงมุมซ้ายบนบอกจำนวนรายการที่ยัง `pending` อยู่ ให้เห็นได้ทันทีว่าโต๊ะไหนครัวยังไม่รับออเดอร์
5. ลูกค้าเองก็เห็นสถานะออเดอร์ตัวเองในหน้าสแกน QR ด้วย (ผ่าน `get_table_order`) จะได้รู้ว่าครัวรับออเดอร์หรือเสิร์ฟหรือยัง
6. แก้ไขสถานะได้เฉพาะบิลที่ยังเปิดอยู่ (`status = 'open'`) เท่านั้น และเฉพาะพนักงานที่ล็อกอินแล้วเท่านั้นที่เรียกฟังก์ชันนี้ได้ ลูกค้าปิดสถานะออเดอร์ตัวเองไม่ได้

### ระบบโปรโมชั่นทำงานอย่างไร

1. เจ้าของร้านเข้าหน้า "โปรโมชั่น" (เมนูเห็นเฉพาะ owner) → กด "เพิ่มโปรโมชั่น" → ตอนนี้รองรับแบบเดียวคือ **"ครบ N ชิ้น ลดตายตัว X บาท"** — ตั้งชื่อ, จำนวนชิ้นต่อรอบส่วนลด, จำนวนเงินที่ลดต่อรอบ, และ **เลือกเมนูที่เข้าร่วมเอง** (อย่างน้อย 1 รายการ) เปิด/ปิดใช้งาน/แก้ไข/ลบได้ตลอด
2. ทุกครั้งที่มีการสั่ง/แก้ไขออเดอร์ในโหมด "เปิดโต๊ะ" (ทั้งพนักงานคีย์เองและลูกค้าสั่งผ่าน QR) **และตอนคิดเงินในโหมด "ขายด่วน"** ระบบจะเรียก `calculate_promo_discount()` คำนวณส่วนลดใหม่อัตโนมัติจากโปรโมชั่นที่เปิดใช้งานอยู่ทั้งหมด แล้วอัปเดต `sales.discount`/`sales.total` ทันที — ฝั่งขายด่วนจะบวกรวมกับส่วนลดที่พนักงานกรอกเองด้วย (ไม่ทับกัน) และหน้าคิดเงินจะโชว์ส่วนลดโปรโมชั่นแยกให้เห็นก่อนยืนยันจ่ายเงินเสมอ
3. วิธีคำนวณ (type `buy_x_get_fixed_discount`): นับจำนวนชิ้น **เฉพาะเมนูที่เลือกไว้ในโปรโมชั่นนั้น** (ไม่นับสินค้าอื่นในบิล) รวมทั้งบิลของโต๊ะ หารด้วยจำนวนชิ้นต่อรอบที่ตั้งไว้ (ปัดลง) แล้วคูณด้วยจำนวนเงินที่ตั้งไว้ต่อรอบ — เช่นเลือกเฉพาะเมนู A/B/C ตั้งไว้ 10 ชิ้น/รอบ ลด 10 บาท/รอบ สั่ง A+B+C รวมครบ 23 ชิ้น = ลด 2 รอบ = 20 บาท — สินค้านอกกลุ่มที่เลือกไว้ไม่ถูกนับเลย
4. เก็บเมนูที่เข้าร่วมไว้ในตาราง `promotion_products` (เจ้าของร้านจัดการเท่านั้น) — โปรโมชั่นที่ยังไม่ได้เลือกเมนูจะไม่มีผลลดใดๆ (ป้องกันตั้งค่าพลาดแล้วลดทั้งบิลโดยไม่ตั้งใจ)
5. หน้าโต๊ะฝั่งพนักงานเห็นส่วนลดที่คำนวณได้ทันทีในหัวข้อโต๊ะและก่อนกด "เก็บเงิน / ปิดโต๊ะ" — ช่อง "ส่วนลด" ในหน้าเก็บเงินจะกรอกส่วนลดโปรโมชั่นให้อัตโนมัติ (แก้เพิ่มเองได้ถ้าต้องการให้ส่วนลดพิเศษเพิ่ม)
6. หน้าลูกค้าสแกน QR จะมี **banner โปรโมชั่นที่เปิดใช้งานอยู่** พร้อมรายชื่อเมนูที่ร่วมรายการให้เห็นตลอดตอนสั่งอาหาร และเมื่อสั่งครบเงื่อนไข จะเห็นส่วนลดที่ตัวเองได้รับในสรุปยอดออเดอร์ทันที (ผ่าน `get_table_order`)
7. ถ้าปิดใช้งานหรือลบโปรโมชั่น บิลที่เปิดอยู่จะคำนวณส่วนลดใหม่ (เป็น 0) ในการสั่ง/แก้ไขรายการครั้งถัดไปโดยอัตโนมัติ

### เงินสดหน้าร้าน (เงินทอน) และเงินโอน ทำงานอย่างไร

1. พนักงาน/เจ้าของร้านเข้าหน้า "เงินสด / เงินโอน" → กด "เปิดกะ" → กรอกเงินสดตั้งต้นที่มีอยู่ในลิ้นชักตอนนี้ (ค่าเริ่มต้นดึงมาจากยอดปิดกะครั้งก่อนให้อัตโนมัติ) — เปิดพร้อมกันได้แค่ 1 กะเท่านั้น (บังคับด้วย unique index ในฐานข้อมูล)
2. ระหว่างกะเปิดอยู่ หน้าเว็บจะโชว์ **"เงินสดที่ควรมีตอนนี้"** แบบสดๆ = เงินตั้งต้น + ยอดขายที่จ่ายด้วยเงินสด (`payment_method = 'cash'`) − รายจ่ายที่จ่ายด้วยเงินสด (`expenses.payment_method = 'cash'`) + ยอดที่ปรับระหว่างกะ นับตั้งแต่เวลาที่เปิดกะ
3. กดปุ่ม **"ปรับยอด"** ได้ตลอดเวลาที่กะเปิดอยู่ (เช่น หยิบเงินไปฝากธนาคาร, นับผิดตอนเปิดกะ) — ต้องระบุเหตุผลเสมอ ยอดที่ปรับจะรวมเข้ายอดคาดการณ์ทันทีและมีผลตอนปิดกะด้วย
4. กด "ปิดกะ / นับเงิน" → กรอกจำนวนเงินสดที่นับได้จริง → ระบบ (`close_cash_shift`) จะคำนวณยอดคาดการณ์ใหม่อีกครั้งฝั่งเซิร์ฟเวอร์ (ไม่เชื่อค่าที่ฝั่งเว็บคำนวณไว้) แล้วโชว์ส่วนต่างขาด/เกินทันที
5. ยอดที่นับได้ตอนปิดกะจะกลายเป็นเงินตั้งต้นที่แนะนำของกะถัดไปโดยอัตโนมัติ (ตอบโจทย์ "เงินที่เหลือใช้เป็นเงินทอนต่อไป")
6. มีประวัติการปิดกะย้อนหลังให้ดู พร้อมป้ายสถานะเขียว/แดง/น้ำเงินบอกว่าตรง/ขาด/เกิน
7. **เงินโอน** แยกเป็นอีกส่วนหนึ่งในหน้าเดียวกัน ไม่ต้องเปิด/ปิดกะ (เพราะไม่ต้องนับสดจริง) — ยอดคำนวณสะสมตลอดกาล = ยอดขายที่จ่ายผ่านโอน − รายจ่ายที่จ่ายผ่านโอน + ยอดที่ปรับเอง กด "ปรับยอด" แก้ไขได้ตลอดเวลาเหมือนเงินสด (ต้องระบุเหตุผลเช่นกัน)

### ประกาศถึงพนักงานทำงานอย่างไร

1. เจ้าของร้านเข้าหน้า "ประกาศ" (เมนูเห็นเฉพาะ owner) → พิมพ์ข้อความ (ไม่เกิน 500 ตัวอักษร) → กด "ส่งประกาศ"
2. พนักงาน/ผู้จัดการ/เจ้าของร้านคนอื่นที่ **เปิดแอปอยู่ตอนนั้น** จะเห็นข้อความขึ้นเป็นกล่องกลางหน้าจอทันทีผ่าน Supabase Realtime (ไม่ต้องรีเฟรชหน้า) พร้อมเสียงเตือน (ทำนองแยกจากเสียงออเดอร์/เรียกเก็บเงิน ฟังออกว่าต่างกัน) กด "รับทราบ / ปิด" เพื่อปิดได้เอง
3. ใครที่ยังไม่ได้เปิดแอปตอนนั้นจะเห็นประกาศล่าสุดที่ยังไม่เคยปิด ทันทีที่เปิดแอปครั้งถัดไป (เก็บสถานะ "ปิดแล้ว" แยกเป็นรายอุปกรณ์/เบราว์เซอร์ผ่าน `localStorage` ไม่ใช่ระดับบัญชีผู้ใช้ — สลับเครื่องจะเห็นซ้ำได้)
4. ถ้าใครตั้งค่า Web Push ไว้ (ดูหัวข้อ "ตั้งค่าแจ้งเตือนแม้ปิดหน้าจอ") จะได้รับแจ้งเตือนแบบ OS-level ด้วยแม้ปิดแท็บ/ล็อกจอ/สลับแอปอยู่ — ใช้กลไก `trigger_push_notify()` เดียวกับออเดอร์/เรียกเก็บเงิน
5. มีประวัติประกาศที่ผ่านมาให้ดูและลบได้ในหน้าเดียวกัน (ลบแล้วพนักงานที่ยังไม่เคยปิดจะไม่เห็นข้อความนั้นอีก)

### ล็อกอินด้วยเบอร์โทรทำงานอย่างไร

Supabase Auth ผูกบัญชีด้วยอีเมลเป็นหลัก เบอร์โทรในระบบนี้จึงเป็น "ทางลัด" ไปหาอีเมลจริงเบื้องหลัง ไม่ใช่ระบบ SMS OTP แยกต่างหาก (ไม่มีค่าใช้จ่ายส่ง SMS):

1. เจ้าของร้านสร้างพนักงานที่หน้า "พนักงาน" — ถ้าไม่กรอกอีเมล ระบบจะสร้างอีเมลปลอมให้อัตโนมัติ (เช่น `phone-0812345678@staff.pos.local`) เพื่อให้ Supabase Auth ใช้งานได้ พนักงานไม่จำเป็นต้องรู้อีเมลนี้เลย
2. ตอนล็อกอิน ถ้าพิมพ์เบอร์โทร (ไม่มี `@`) หน้า login จะเรียก `get_email_by_phone()` หาอีเมลจริงก่อน แล้วค่อย sign in ด้วยอีเมล+รหัสผ่านตามปกติ
3. บัญชีที่ไม่มีอีเมลจริงจะ **รีเซ็ตรหัสผ่านผ่านอีเมลไม่ได้** — เจ้าของร้านต้องกด "เปลี่ยนรหัสผ่าน" ให้ที่หน้า "พนักงาน" แทน

|---|---|---|---|
| ขายหน้าร้าน (สร้างบิล) | ✅ | ✅ | ✅ |
| ประวัติการขาย, ยกเลิกบิล | ✅ | ✅ | ✅ |
| แก้ไข/ลบสินค้า/หมวดหมู่ | ✅ | ✅ | ✅ |
| ภาพรวมร้าน (Dashboard) | ❌ | ❌ | ✅ |
| รายจ่าย, รายได้ | ❌ | ❌ | ✅ |
| จัดการ role พนักงาน | ❌ | ❌ | ✅ |
| ดู Log กิจกรรม | ❌ | ❌ | ✅ |

> staff และ manager มีสิทธิ์เท่ากันทุกอย่างในตอนนี้ (เห็นแค่ขายสินค้า/ประวัติขาย/สินค้า) — เผื่อไว้สำหรับแยกสิทธิ์เพิ่มเติมในอนาคตถ้าต้องการ
